# Haiku Classification Prompt Tuning — Design Spec

**Date:** 2026-03-23
**Goal:** Optimize the AI classification prompt by running Haiku inside Claude Code, comparing against ground truth, interviewing failures, and generating rule candidates.

---

## 1. Problem Statement

The CMA classification pipeline's Tier 2 (AI Classifier) uses a prompt in `backend/app/services/classification/ai_classifier.py:_build_prompt()` to classify financial line items into 139 CMA fields. Current issues:

- **53.4% doubt rate** on BCIPL (369/691 items flagged as uncertain)
- **API cost** — every item that reaches Tier 2 costs OpenRouter tokens
- **No feedback loop** — prompt was written once, never tuned against real data
- **Section context underused** — "Interest Received" from Notes to P&L (row 30) vs Notes to BS (row 247) depends on section, but the prompt doesn't emphasize this

## 3. Input Data

### 3a. Ground Truth
- **File:** `DOCS/extractions/BCIPL_classification_ground_truth.json` (being generated by Sonnet agent)
- **Format:** Array of objects, each with:
  - `raw_text` — the line item description as extracted
  - `amount_rupees` — the amount in absolute rupees
  - `financial_year` — 2021, 2022, or 2023
  - `section` — which sheet/section it came from (profit_and_loss, balance_sheet, notes_to_pl, notes_to_bs, depreciation_schedule, etc.)
  - `correct_cma_field` — the correct CMA field name (exact match from `cma_field_rows.py`)
  - `correct_cma_row` — the correct row number in INPUT SHEET
  - `confidence_note` — "verified", "high", or "inferred"

### 3b. Item Sources (sheets per FY)
Each BCIPL financial year Excel contains:
- Notes to P&L — expense breakdowns (wages, raw materials, each line)
- Notes to BS — asset/liability breakdowns (receivables, payables, loans)
- Subnotes to PL — sub-breakdowns (revenue by type, interest by type)
- Subnotes to BS — sub-breakdowns (inventory types, advance types)
- Depreciation Schedule — asset-wise depreciation (Plant, Vehicle, Furniture, etc.)
- P&L summary, BS summary (usually subtotals — may skip)
- Trial Balance — EXCLUDED by extractor

### 3c. Current Prompt
The exact prompt from `ai_classifier.py:453-478` will be used as the **baseline**. It includes:
- Line item description, amount, section, industry type, document type
- Fuzzy match suggestions (top 5 candidates)
- All 139 valid CMA fields with row numbers
- 7 classification rules

## 4. Phases

### Phase 1: Prepare (Opus, no Haiku)
1. Read `BCIPL_classification_ground_truth.json`
2. Deduplicate: group by `raw_text` (case-insensitive), keep one representative per unique item
3. Tag each item with its `section` context (critical for routing)
4. Include depreciation schedule items explicitly
5. Expected: 200-250 unique items
6. Split into batches of 15

### Phase 2: Baseline Classification (Haiku agents)
For each batch of 15 items, spawn a Haiku agent with this prompt structure:

```
You are testing a classification prompt. For each line item below,
classify it into a CMA field using ONLY the prompt provided.

[THE EXACT _build_prompt() OUTPUT FOR ITEM 1]
---
[THE EXACT _build_prompt() OUTPUT FOR ITEM 2]
---
... (15 items)

Return a JSON array with your classification for each item:
[
  {
    "item_index": 0,
    "raw_text": "...",
    "classified_cma_field": "...",
    "classified_cma_row": ...,
    "confidence": ...,
    "reasoning": "..."
  },
  ...
]
```

**Token tracking:** For each batch, record:
- Approximate input token count (prompt length)
- Output token count (response length)

### Phase 3: Failure Analysis (Opus, no Haiku)
1. Compare each Haiku response against ground truth
2. Score: correct if `classified_cma_row == correct_cma_row`
3. Group failures by pattern:
   - **Section confusion** — item exists in multiple sections, wrong one picked
   - **Field proximity** — correct field is adjacent/similar (e.g., R41 vs R42, imported vs indigenous)
   - **Depreciation routing** — manufacturing vs admin depreciation
   - **Interest routing** — term loan vs working capital vs bank charges
   - **Freight/transport** — manufacturing vs selling expense
   - **Statutory dues** — which liability row
   - **Catch-all overflow** — items dumped into "Others" rows when a specific row exists
4. Calculate baseline accuracy: X/Y correct (Z%)
5. Estimate cost: total tokens × OpenRouter rate ($0.80/$4.00 per M tokens for Haiku)

### Phase 4: Pattern Interview (Haiku agents)
For each failure pattern (3-6 patterns expected), spawn one Haiku agent:

```
You are a financial analyst who just classified these CMA line items.
You got them WRONG. I want to understand why.

FAILURE PATTERN: [e.g., "Depreciation routing"]

ITEMS YOU MISCLASSIFIED:
1. "Depreciation on Plant & Machinery" — You said: R63 (Depreciation CMA/Admin)
   Correct answer: R56 (Depreciation Manufacturing)

2. "Depreciation on Vehicles" — You said: R56 (Depreciation Manufacturing)
   Correct answer: R63 (Depreciation CMA/Admin)

... (up to 10 examples per pattern)

QUESTIONS:
1. What in the prompt made you choose the wrong field?
2. What additional context would have helped you get it right?
3. Is there a simple rule that covers ALL these items?
   (e.g., "If asset is used in production → R56, otherwise → R63")
4. Would knowing the industry type (Manufacturing) change your answer?
5. Should the prompt group related CMA fields together instead of
   listing all 139 as a flat list?
```

### Phase 5: Prompt Revision (Opus, no Haiku)
Based on interview responses, revise the prompt. Potential improvements:
- Add **section-aware routing hints** ("items from Notes to BS are liabilities/assets, not expenses")
- Add **industry-specific guidance** ("for Manufacturing: freight → R47, depreciation on production assets → R56")
- **Group CMA fields by category** instead of flat list (P&L Income, P&L Manufacturing, P&L Admin, BS Assets, BS Liabilities)
- Add **common confusion pairs** ("R41 Imported vs R42 Indigenous — check if description says 'import' or 'customs'")
- Reduce the CMA field list to only relevant fields based on section context

### Phase 6: Re-test (Haiku agents)
1. Send ONLY previously-failed items with the revised prompt
2. Compare results
3. Calculate improved accuracy
4. If accuracy gain < 5%, iterate phases 4-6 with remaining failures
5. Max 3 iterations total

## 5. Constraints

1. **Max agent spawns:** ~30 total across all phases (15-17 classify + 4-6 interview + 3-5 re-test)
2. **No code modification:** This is a READ-ONLY experiment. We don't change app code.
3. **No OpenRouter cost:** Everything runs on Claude Max subscription via Claude Code agents
4. **Max 3 prompt revision iterations** — if accuracy plateaus, stop and report
5. **Ground truth items with `confidence_note="unknown"` are excluded** from accuracy calculation
6. **Batch size: 15 items per Haiku agent** — balances throughput vs context quality
7. **All results saved to:** `DOCS/test-results/bcipl/prompt-tuning-2026-03-23/`

## 7. Success Criteria

| Metric | Current (estimated) | Target |
|--------|-------------------|--------|
| Classification accuracy | ~50-60% | 80%+ |
| Doubt rate | 53.4% | <25% |
| New rule candidates | 0 | 15+ |
| Cost per item (tokens) | Unknown | Measured |

## 8. Risks

| Risk | Mitigation |
|------|-----------|
| Ground truth JSON not ready | Wait for Sonnet agent to complete before starting Phase 2 |
| Haiku performs differently in batch vs single-item | Use exact same prompt format as app; validate 5 items individually |
| Too many unique items (>300) | Sample 200 most representative, full test in separate session |
| Interview responses are vague | Ask specific questions, provide concrete examples |
| Max subscription token limits | Monitor agent count, stop if approaching limits |
